Configuration and Status

The AEVI SDK includes a number of APIs and Helpers to assist the Developer in building apps that are aware of the environment in which they operate.

Configuration APIs

The Configuration APIs enables the app developer to obtain information about Albert and the installed Payment App including the terminal id, API version, whether the app is talking to the Simulator or runs on the AEVI Emulator.

Permissions

In order to interact with the Configuration API, each app needs to include the following permission in its Manifest file.

<uses-permission android:name="com.aevi.permission.CONFIGURATION" />

For more information about the available permissions, please see Manifest section.

Obtaining Terminal details

The TerminalConfiguration object contains information about the configuration of Albert the app is running on. Obtaining the terminal configuration is very easy :

// Get Terminal configuration
TerminalConfiguration configuration = TerminalConfiguration.getTerminalConfiguration(this);
String terminalId = configuration.getTerminalId();

The following information is available on the Terminal Configuration object :

Name Type Description
ApiVersionMajor int The major part of the version number. Until the official release this will be 0
ApiVersionMinor int The minor part of the version number.
ApiVersionBugFix int The bug fix part of the version number.
Version String The complete version number comprised of the major,minor and bug fix number.
TerminalId String A unique ID which can be used in identifying the terminal
Simulator boolean Indicates if the app talks to the Simulator or actual Payment Application
AEVIDevice boolean Indicates if it is a real terminal or the AEVI Emulator

PaymentAppConfiguration

The PaymentAppConfiguration forms part of the Payment API and provides information which is relevant for the payment process such as: the default currency; the supported transactions; and the payment app state.

Permissions

In order to interact with the PaymentAppConfiguration API, each app needs to include the follow permission in its Manifest file.

<uses-permission android:name="com.aevi.permission.CONFIGURATION" />

For more information about the available permissions, please see Manifest section.

Obtaining Payment App Details

The PaymentAppConfiguration object can be used to query the capabilities and configuration settings. Before doing so it is considered a good idea to ensure that a Payment app is installed using the getPaymentApplicationStatus method on the StatusService. For example, from inside an Activity one could write:

/**
 * Get the PaymentAppConfiguration
 */
private void getPaymentAppConfiguration() {
  if (PaymentAppConfiguration.getPaymentApplicationStatus(this) == ServiceState.AVAILABLE) {
     // Get Payment app configuration
    startActivityForResult(PaymentAppConfigurationRequest.createIntent(), 0);
  }
}

/**
 * Called by Android when the control returns to this activity.
 *
 * @param requestCode the code associated with the request (0)
 * @param resultCode  the Intent result code
 * @param data        the result data
 */
@Override
protected void onActivityResult(int requestCode, int resultCode, Intent data) {
  if (requestCode == 0 && resultCode == Activity.RESULT_OK) {

    // Save the Payment Configuration in the Application Object
    PaymentAppConfiguration paymentAppConfiguration = PaymentAppConfiguration.fromIntent(data);
    XYZApplication xyzApplication = (XYZApplication) getApplication();
    xyzApplication.setPaymentAppConfiguration(paymentAppConfiguration);
  }
}

Please note certain attributes of the PaymentAppConfiguration object are volatile (e.g. ApplicationState) as they contain the state of the Payment app at the time of query. This information should be considered volatile since the Payment app may change its state or configuration at any time, thus invalidating any stored or cached copies of this object.

The following information is available from the Payment Configuration object :

Name Type Description
Vendor String Payment app vendor
Version String Payment app version
SupportedCurrencies Currency[] List of supported currencies
DefaultCurrency Currency The default currency
Merchants MerchantInfo[] List of available Merchants
SupportedTransactions TransactionType[] List of supported transactions
ApplicationState ApplicationState State of the payment app, is either Available or Busy
IntegratedMode boolean True if device is connected to an external Cash Register or POS System
CanPrintCombinedReceipt boolean True if this payment application is capable of printing a single combined receipt

As listed above the ApplicationState indicates if the payment app is available. If this is not the case then the payment app will not be able to handle a request. For an app developer it is sensible to check the availability before performing a request. When the payment app is busy the app can wait a certain time and try again or terminate the request.

The Albert device can be used as a stand-alone payment device or integrated with an external Cash Register or POS System. The IntegratedMode flag in the table above allows for inspection of the current integrated status. It is possible that this status may change during device operation, for instance if the external POS System is disconnected or disabled.

Even when the payment app has indicated that it is available it might not be so at the moment of the request. In that case an exception will be thrown.

Service State

A number of the AEVI APIs are implemented as Android Services. Examples are the Print Service, the Crashlog Service and the Card Reading Service.

It is possible to query service state by using the TerminalConfiguration class. This class includes methods to indicate if a service is installed, if the calling app has permission for the service and if it is generally available.

The possible resulting states are:

NO_PERMISSION :the calling app does not have the correct permissions to perform this action

NOT_INSTALLED :the requested service is not installed on Albert

UNAVAILABLE :the requested service is installed on Albert but in a fault state

AVAILABLE. :the requested service is installed on Albert and ready to be used

Service State Example

The availability of the services can be checked individually as shown in the example below for the print service:

if (TerminalConfiguration.getServiceStatus(PrintService.class, this) != ServiceState.AVAILABLE) {
   //Handle unavailability of the printer service
}

API Compatibility and Provided Functionality

The APIs provided with the AEVI SDK are designed to be ‘vendor independent’. As an example, this ensures that an app written for one bank also runs on a device supplied by other banks. This design philosophy can result in some APIs being either unique to, or alternatively, unsupported for specific platforms. Next to this vendor agnostic approach, services on the platform are also likely to be updated with a different pace than individual apps.

These properties can potentially lead to a situation where platform services differ in both version and function to an API library statically linked inside an app. Although this might not be a problem per-se it is strongly recommended to assure that the APIs consumed are (still) compatible and supported on the platform the app run on.

To determine this compatibility the TerminalConfiguration and PaymentAppConfiguration APIs provide additional functionality to ascertain version information for each individual component and indicates the availability of an API on the current device. Based on this information an app can, for instance, disable or enable non-critical functions of the application.

In the example below the compatibly information is queried and the mail function is checked for both support and version compatibility. The function isMailApiUsable returns a boolean indicating if the mail API is usable on the device and getMailApiStatus returns a set of impediments for using the API. If there are no impediments it returns an empty set.

// get TerminalConfiguration
TerminalConfiguration terminalConfiguration = TerminalConfiguration.getTerminalConfiguration(this);

// test mail support & compatibility
if (terminalConfiguration.isMailApiUsable()) {
  // enable mail receipt function
} else {
  for (ApiStatus status : terminalConfiguration.getMailApiStatus()){
  // report the impediments of the service
    Log.e(TAG,"Mail API is unavailable. Reason :"+status);
  }
}

Methods exist to allow the developer to determine the status of all SDK APIs in a similar manner to the example above. Both the TerminalConfiguration and PaymentAppConfiguration provide methods to check the individual API services and if they are usable or not. These methods all follow the same structure as the email example above. See the TerminalConfiguration and PaymentAppConfiguration javadocs for more details.

Compatibility Exception

Applications compiled using a newer version of the SDK library should not be installed ahead of time on an Albert that has not yet been upgraded to supported these new features. If this happens then a CompatibilityException will be thrown.

The following rules define when this exception will be thrown. The version number from the AEVI SDK library used by the application is compared with the Device API version number and also with the Payment API version number that is installed on the Albert. If the AEVI SDK library version number is found to be newer than either of these then a CompatibilityException is thrown. The SDK uses the Semantic Versioning Specification (SemVer) with the major and minor parts of the version number used for this comparison. The patch part of the version number is not used for this comparison.

Integrated Mode Broadcast Message

When the Integrated Mode of the device is changed a broadcast message is sent. You can listen for these broadcast messages by registering programatically or adding your BroadcastReceiver to the android manifest of your application.

Your requirements should determine which of these methods you use within your application but it is usually only necessary to use one of the methods to listen for the broadcast Intent. If your application is required to react to changes in Integrated mode without being in the foreground then the BroadcastReceiver should be registered in the manifest. If however you are only concerned with the integrated mode whilst you application is in the foreground then you should consider registering it within the code of your Activity.

The AEVISample application registers its BroadcastReceiver within its Android manifest and displays a Toast message when the status changes.

In both cases it will be necessary to create a BroadcastReceiver to receive the broadcast Intent e.g.

public class IntegrationModeReceiver extends BroadcastReceiver {

    public static final String TAG = IntegrationModeReceiver.class.getSimpleName();

    @Override
    public void onReceive(Context context, Intent intent) {
        boolean intMode = intent.getBooleanExtra(PaymentAppConfiguration.EXTRA_INTEGRATED_MODE, false);
        Log.i(TAG, "Received integration mode broadcast event. Value: " + intMode);
    }
}

To register within an activity to receive the broadcast events programmatically you would then place the following code somewhere within your Activity. Usually in the onResume() method.

IntentFilter filter = new IntentFilter(PaymentAppConfiguration.ACTION_BROADCAST_INTEGRATED_MODE);
IntegrationModeReceiver intModeReceiver = new IntegrationModeReceiver();
registerReceiver(intModeReceiver, filter);

It is important to ensure that you also unregister for these events when your Activity is no longer visible, usually this is done in the onPause() method of your activity if you registered the receiver in the the onResume() method, or within the onDestroy() method if you registered your receiver in the onCreate() method of your activity.

@Override
     protected void onDestroy() {
         super.onDestroy();

         if(intModeReceiver != null) {
             unregisterReceiver(intModeReceiver);
         }
     }

If you wish to receive the broadcast events while your application is closed then you can add details of your receiver to the Android manifest.

<application  >
    ...
    <receiver android:name=".IntegrationModeReceiver">
        <intent-filter>
            <action android:name="com.aevi.payment.BROADCAST_INTEGRATION_MODE"></action>
        </intent-filter>
    </receiver>
    ...
</application>
