Permissions and Manifest changes

Quick Index

  • Supported Android Permissions

The platform is a privilege-separated environment, in which each app runs with a distinct system identity and app group. Finer grained features are provided through a permission mechanism that enforces restrictions on the specific operations that a particular process can perform. This chapter comprises the following sections:

Permissions

By default, a basic app has no permissions associated with it, therefore it cannot adversely impact the user experience or any data on Albert. To make use of protected features of Albert, include in the AndroidManifest.xml one or more <uses-permission> tags declaring the permissions that your app needs.

The following permissions are available:

During the app installation process, the requested permissions are inspected and granted.

Example manifest

The following app requests permission to access the payment app and send a payment request.

<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.mycompany.app.myapp" >
    <uses-permission android:name="com.aevi.permission.CONFIGURATION" />
    <uses-permission android:name="com.aevi.permission.PAYMENT_REQUEST" />
    ...
</manifest>

Type Enforcement

The Type Enforcement system imposes a mandatory and strict separation of system processes according to least privilege. In addition, the system applies strict separation of apps from the Firmware. The solution is based on SELinux and SEAndroid.

Type Enforcement has also been extended to impose separation between different types of Apps. For app developers, the following types of app are available:

Application Type Description Manifest Setting
Network Apps that do not need to use the AEVI SDK APIs vaa_network
Merchant Apps that require access to any of the AEVI SDK Services vaa_merchant

To indicate the group type to which an app belongs, an <application-type> tag must be included in the AndroidManifest.xml file. The structure of the entry is as follows:

<meta-data android:name="application-type" android:value=["vaa_network"|"vaa_merchant"] />

For more details on Type Enforcement, please read the Platform Hardening & Restrictions chapter.

Note: Apps that do not have anapplication-type tag will not run. And only those apps of type vaa_merchant will have access to AEVI SDK API.

Example manifest

The following app requests permission to access the Payment app.

<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.mycompany.app.myapp" >
    ...
    </application>
    <meta-data android:name="application-type" android:value="vaa_merchant"/>
</manifest>

Network access

For standard Android, basic networking policy is controlled by the permissions in the application manifest, there are two network access levels defined by Android:

  1. Apps that have the internet permission defined have full access to the network
  2. Apps that do not have the internet permission have no network access

In addition, Albert’s Trusted Communications Manager (TCM) uses iptables to implement a System-Wide enforcement of the network security policy. Therefore Apps that require network access must also declare the endpoints to which they connect within the Android Manifest. During App installation, these rules are extracted and the iptables are updated.

Network connections for apps are made using the normal Android HTTP APIs. For Albert, these behave in the same way and throw the same exceptions as on standard Android.

Configuring the AndroidManifest.xml file

The ip-addresses section of the AndroidManifest.xml is structured as follows:

<meta-data android:name="ip-addresses" android:value="string" />

The content in the string is defined as follows:

<IP address> or <host name>

The <IP address> entry may refer to the destination IP address or IP address range that should be whitelisted for the specified application. It is possible to use IP address ranges for the <IP address> entry of the form

192.168.1.0/24 results in an IP address range from 192.168.1.1 to 192.168.1.255 
10.0.0.0/8 results in an IP address range from 10.0.0.1 to 10.255.255.255
10.0.0.1,10.0.0.3 results in an IP address range from 10.0.0.1 and 10.0.0.3
10.0.1.15-10.0.2.5 results in an IP address range from 10.0.1.15 to 10.0.2.5

An IP address range is limited to 255 continuous addresses. An incorrectly specified IP-Address range will be ignored resulting in no connectivity. It is possible to combine ranges and single IP addresses. These are separated by commas as in the following example:

192.168.1.0/24,192.168.2.10-192.168.2.90,192.168.3.7

The following wildcard can be used if no restrictions are desired for a specific app.

0.0.0.0/0

As an example, the manifest file would be configured for full Internet access in the following manner:

<meta-data android:name="ip-addresses" android:value="0.0.0.0/0" />

Note: Wildcard settings will require Bank approval.

LAN Communication

For LAN communication, the ip-addresses entry may be configured with the specific whitelist string, ‘local’. The manifest file should be configured as follows:

<meta-data android:name="ip-addresses" android:value="local" />

Note: IPv6 is currently not supported.

Declaring multiple discrete endpoints and the Gradle build tool

It is possible to declare multiple discrete endpoints by declaring multiple meta-data tags. However, if the Gradle build tool is used, it is likely that you will get errors during the build process. In this scenario, you are required to add a suffix to the ip-addresses attribute as per the following example:

<meta-data android:name="ip-addresses01" android:value="api.example.com" />
<meta-data android:name="ip-addresses02" android:value="cdn.example.com" />
