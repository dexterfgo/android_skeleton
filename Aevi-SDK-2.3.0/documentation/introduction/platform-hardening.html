In order to create a secure device that can be approved for use as a Payment Terminal, we have had to harden and restrict the core Android platform. This section provides information on these modifications and the restrictions carried out.

Android Hardening

A number of complementary measures have been undertaken to secure the Albert Platform from a Software and Firmware perspective:

  • Albert runs on the Security Enhanced version of Android (SEAndroid)
  • The Kernel and System have been hardened
  • All apps installed on Albert are Type Enforced - restricting their access to Services needed
  • A Trusted Communications Manager (TCM) has been implemented - this enforces the requirement for declarative access to Network Services via an App’s Android Manifest
  • Albert includes a firewall to control network traffic
  • All App Install files (APKs) must be signed by the Bank prior to installation
  • The Payment App runs in a hardened Sandbox. Access to the Payment App is restricted to a limited set of APIs

Security Enhanced Android

In the OS, we have applied SEAndroid patches to the kernel and system services in order to be able to enforce mandatory Type Enforcement (TE) security policies.

For further information please visit Security Enhancements for Android.

Kernel & System Hardening

A number of mechanisms have been introduced to harden the kernel and system. These typically include isolation mechanisms and follow a ‘least privilege’ principle.

  • Privileges of system services have been reduced
  • Type Enforcement policies have been defined that restrict the system services access only to the resources they need
  • Redundant modules (such as device drivers) have been removed from the kernel
  • Additional system security services have been introduced. These include an UpdateManager which is responsible for managing Firmware updates. It also includes a Trusted Communications Manager (TCM) which is responsible for controlling Network access per app
  • Mandatory type enforcement for all apps and services
  • Insecure default configuration options have been removed
  • Installation of Android apps by non-authorised services is prevented
  • Access to critical/internal storage is disabled (an example of a critical/internal storage is: /dev/kmem)
  • Access to sensitive file locations is prohibited
  • Apps are prevented from accessing data directories of other apps
  • Android Debug Bridge is not available on production devices
  • Hardware devices are mapped to virtual devices. Type enforcement is used to control access to virtual devices
  • The code complexity of system services has been reduced by removing non-relevant library references and by using hardened library code
  • Network tethering is disabled
  • Access to the Internet is disabled for USB & Bluetooth Modems. These devices are limited to supporting Payment transaction processing.

Type Enforcement

The Type Enforcement system imposes a mandatory and strict separation of system processes, according to least privilege. In addition, the system also imposes strict separation of apps from the firmware. The solution is based on SELinux and SEAndroid. Type Enforcement has also been extended to enforce separation between different types of apps. For App developers, the following types are available:

  • Merchant - apps that require access to the AEVI SDK Services
  • Network - apps without any requirement for access to AEVI SDK Services

The Type Enforcement Mechanism also limits inter-app communication between app types. For example, whilst a Network app can call another Network app, a Network app cannot call a Merchant app. The converse is also true; a Payment Relevant app can call a Payment Relevant but cannot call a Network App.

All apps must define the app type in the Android Manifest. Please see the Manifest section.

Network restrictions

Trusted Communications Manager

As with standard Android, the basic network policy for Apps on Albert is controlled by standard Android permissions defined in the application manifest. In summary there are two network access levels defined by Android:

  • Apps that have the internet permission defined have full access to the network
  • Apps that do not have the internet permission have no network access

In addition, access to the network is further controlled by the Trusted Communications Manager (TCM) which uses iptables to implement a System-Wide enforcement of the network security policy; iptables rules are embedded in the apps Android Manifest.

In summary, Apps are required to declare the domains and/or the IP addresses to which they connect within the App Android Manifest. During installation, these rules are extracted and the iptables updated. Note that, subject to Bank approval, wildcards can be used for IP Addresses and Domains within the Manifest.

Please see the Network Access section for configuration details.

Bank imposed restrictions

Please note that the Bank may impose network bandwidth restrictions on Apps when Albert is connected via a 3G Modem. Further information will be made available on the Developer Portal in due course.

App Signing

The Operating System has been modified to verify the Signature of all apps installed, ensuring they have been signed by the Bank. This means that all apps will require approval by the Bank. Please refer to the Developer Portal for further information.

Launcher App Icon Size

The Albert has a high-density screen and uses a custom home screen launcher with double the normal grid spacing. As a result the app launcher icons need to be double the linear size of a standard Android launcher icon.

Please ensure your App icons have a size of 144x144 pixels and place them in the drawable-hdpi folder.

Available Apps

Many of the typical stock Android tablets and phones include access to other apps such as Google Maps, Google Chrome and Gmail. These are not currently offered on Albert. The following apps are available on the standard device image;

Customisations

Several customisations where made to the Android platform to run on the Albert hardware. The following section lists the one that are of interest for developers.

System Bar

The standard Android System Bar has been customised as part of the development of Albert:

  1. The System Bar has been placed at the top of the screen
  2. The icons have been customised
  3. The Navigation Buttons can be programmatically hidden
  4. Interaction with notifications can be programmatically disabled
  5. A Background Task Indicator is available
  6. A Progress Notification can be displayed

Further detail on using these extended features will be made available in a future release of the AEVI SDK.

Lock Screen

As a secure device, Albert requires entry of a screen Unlock Password (see below) both at Boot and when resuming from Standby Mode.

Unlock screen

Albert is configured to support a number of different roles:

  • Operator : For regular Users
  • Manager : For managers who have additional privileges on Albert. Typically used for accessing specific Administration & Configuration functions on Albert
  • Technician : Used by Technicians when configuring or troubleshooting issues

Passwords are typically 4-6 digits long. App developers will also be able to use this mechanism to control access to parts of their app in upcoming releases.

App Launcher

A custom Android Launcher is optionally available to Wincor Nixdorf customers comprising a number of specific features:

  • A larger grid of apps (typically a 4x4 array of icons)
  • App installation and removal is controlled by the Albert Management System
  • Support for multiple pages of apps
  • Support for app re-ordering

The key limitations to be aware of include:

  • The Launcher app cannot be replaced by a 3rd party Launcher
  • Apps cannot be uninstalled from the Launcher Screen
  • There is no support for app Widgets

Please note that, at the current time, launcher are specific to the specific Bank variant of Albert. Banks may provide alternative launchers and may also choose to provide the standard Android launcher that offers full Widget capability.

The Merchant Menu is typically used by the merchant to select app Functionality or Options. It is accessed in one of two ways:

  • Double Finger Swipe Down from the Top of the Screen,

Access to System Settings

The Android settings app is not accessible from the launcher and a bespoke app is provided instead. This customised settings app provides a simpler interface as well as providing access controls to the various settings required for Albert - these use the password mechanism described above.

Other Restrictions & Limitations

Furthermore, Albert is subject to the following restrictions and limitations.

Installation of apps

Apps can only be installed through an offical AEVI app store. There is no access to the Google Play Store (or any other third party app store). Furthermore, all apps must be signed by the Bank or they will be rejected by the Package Manager. The Albert Management System (comprising an Agent on Albert and a Bank hosted service) is responsible for the installation, removal and update of apps on Albert. Whilst the Albert Management System operates in the background, and users never interact with it, it occasionally displays notification messages on the screen to advise of background activity.

Access to Phone and Messaging Services

Whilst Albert includes a 3G Modem and Data Card, there is no Phone or Messaging capability.

Use of the Native Development Kit

Android has published a Native Development Kit (NDK) for the development of apps. Unfortunately we currently do not permit apps to be developed in this technology.

USB Connectivity

USB Host Mode is supported. Therefore, apps can, for example, read USB Flash disks, assuming that the appropriate connectors are obtained. However, this is only available to Payment Relevant apps (see the Type Enforcement section above). Furthermore, when a USB Device is installed, unlike stock

Android, Albert does not display a prompt requesting the user’s permission to connect to the USB device.

Developer Mode

Android Developer Mode is not available on Albert. These have been explicitly disabled and removed as part of the Hardening process.

WebView Widget

The WebView Widget is available and accessible on Albert. However, access to external sites will be governed by the Network permissions granted by your app label under the Type Enforcement rules described above.

Due to a vulnerability identified in Android 4.0, the WebView component has been patched and all functions exposed to JavaScript must include the JavascriptInterface annotation.

Note: This annotation is only supported from Android API Level 17. Therefore, you are required to ensure that your compilation settings are set accordingly and that you have the corresponding build tools installed within your environment.

Adobe Flash

The Adobe Flash Player is not supported on Albert. It is suggested that as an alternative, Developers consider using HTML5, CSS3 and JavaScript. Please refer to the ‘ToTheMovies’ Sample App for an example of this.

No Hardware GPU Acceleration

By default Android uses hardware acceleration to render 2D View’s using the GPU. This feature is not available on the Albert or the AEVI Emulator. To get similar performance on your development tablet you can turn off the hardware acceleration for your entire application by adding the following to the <application> tag in the Manifest file.

<application android:hardwareAccelerated="false" >

Policy Restrictions

App developers need to be aware that there are other restrictions to the platform that are not applied systemically but are applied as part of the app review process. The full list can be found on the Developer Portal.

Access to Google Services

Access to Google Services such as Google Play (all services), Google Cloud Messaging and the Google Maps Android API are not supported. This is an area we are actively addressing. Please watch this space for further developments.

Special Purpose apps

As a general policy, special purpose apps such as launchers, task managers, battery management apps, flashlights, memory cleaners etc. will not be accepted. Please see the Developer Portal for more information.

Web Based Apps

Whilst it is possible to write a Web based (HTML5, CSS3 & JavaScript) App, some Acquirers and Retailers may not allow these types of Apps on their platform for security reasons. Please check with your Acquirer / Retailer.

App Experience

Maintaining a positive experience in the usage of the Platform is a key criteria. The app review process will undertake an extensive review of the app to assess security vulnerabilities, performance and reliability. Apps that fail these tests an extensive review of the app to assess security vulnerabilities, performance and reliability. Apps that fail these tests are likely to be rejected. Please see the Developer Portal for more information.

Interference with Albert or Other apps

Apps that interfere or affect the normal operation of other apps or Albert, including those that reserve excessive amounts of resources (e.g. memory & processor) are likely to be rejected. Please see the Developer Portal for more information.

Impersonation of System or Payment UIs

Apps may not impersonate system or payment related user interfaces. Please see the Developer Portal for more information.

Script Engines and Dynamic Code

Apps that include script engines or those that dynamically download of executable code via the Network are prohibited. Please see the Developer Portal for more information.

Data Capture & Storage

Apps must consider the impact of legislation on the data that is captured by their apps. This includes legislation around the capture and processing of personal data and financial data.

Please see the Developer Portal for more information.

Exposing APIs from your app

Apps must not expose any app programming interfaces of their own. Should you specifically require this type capability, you are advised get in contact via the Developer Portal.

Electronic Cash Register Integration

We recognise that there will be a requirement to integrate Albert with Point of Sale Systems. There is a roadmap to support a number of the Electronic Cash Register (ECR) Standards including PC-EFTPOS, Global Blue and OPI. An API is available for this, however, it is a privileged API that is not be available as part of the AEVI SDK. Please get in contact via the Developer Portal for further information.
