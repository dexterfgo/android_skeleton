AEVI SDK Introduction

We recognise that the ultimate success of this platform dependents on the support of the developer community. Therefore, from the outset, we have prioritised the release of the AEVI Software Development Kit (SDK).

The AEVI SDK for Albert provides you with everything you need to integrate your Android app with the features and functionality of Albert, including the ability for your app to take payments.

AEVI SDK Design Criteria

We defined a number of design criteria to guide us through the development of the AEVI SDK:

  • The AEVI SDK is predominantly for developers to create business apps and utilities that will add commercial value to both the Merchants and Cardholding Customers.
  • We do not expect developers to write or port games to the platform, nor do we expect that merchants will wish to install them.
  • The AEVI SDK should be easy to use by developers with limited experience of payment processes or the banking industry.
  • The AEVI SDK should apply universally across all Banks.
  • The AEVI SDK should be flexible and not prescribe a specific architecture or design for an app.
  • Porting and integrating existing Android apps should be simple and straightforward.
  • Maintaining the integrity of the Platform and the data on the platform is the key. To ensure platform and data integrity, the AEVI SDK does not expose any functionality that might increase the risk to either the platform, or the financial data held by it.
  • The AEVI SDK should be simple and straightforward to use by the majority of developers (further advice to be available through the Developers Portal).
  • The Developer is our most important Customer and we want them to be delighted with the service we provide.

Caution! Not all banks support the same functionality. Before building an app that requires certain features please check the API compatibility table

Documentation Structure

The AEVI SDK Documentation comprises three main volumes: Introduction, Tutorials and Reference. The Introduction volume will help you to familiarise yourself with Albert, the Platform, the Development process and any specific customisation made to Android. The Tutorials aim to familiarise you with the APIs. The Reference Volume is a useful reference for both novices and seasoned developers alike.

Pre-requisites

In order to use the AEVI SDK, you will require some previous experience of developing Java based Android apps together with a suitable development environment, such as:

  • A Computer with the latest version of the Android SDK installed - See Get the Android SDK,
  • A Tablet running Android 4.0.x or later with a screen resolution of 1280x800 (tvdpi). The Google Nexus 7 (2012) is recommended as the screen resolution is identical to that of the Device. (Ensure that Developer mode is enabled and you are able to access the device using the Android Debug Bridge (ADB) utility).
  • An alternative option is to use the emulator that is distributed with the Android SDK. However, we strongly recommend using a generic Tablet for developing and debugging as the Android Emulator can be slow.
  • An Integrated Development Environment (IDE) that supports Android.

Note: Android versions 4.1.x onwards have separated the navigation buttons from notifications resulting in a navigation bar at the bottom of the screen and a notification bar at the top. This differs to Android 4.0.x on which the Albert Device is based. This will have an impact on the amount of vertical screen space your App will have available.

Getting Started

We recommend that all developers start off by installing and trying out the Sample apps. Once you have seen these in action you can review the source code. Alternatively, follow the Getting started with the ‘Payment API’ tutorial.

Having built a simple app that takes payment, read through the rest of the documentation and follow the other tutorials.

API Architecture

Albert comes with a set of apps that enables it to process card based payments. The payment app is responsible for communicating with the bank’s financial host and interfacing with various card schemes and protocols.

An app uses the payment app and other platform services through the payment API. This approach ensures that the apps remain outside the scope of (Payment Card Industry) PCI Accreditation.

Operating scenarios

The Device can run in two different modes:

  • Standalone Mode. Also known as ‘master mode’. In this mode Albert is not connected to any external entity such as an ECR or POS system. When operating in this mode the apps only have limited control of the payment workflow. Apps must specify the payment request, and then the payment app takes control, executes the request and provides a response. The app cannot alter the steps needed by the payment app to fulfil the requested transaction.

  • Integrated Mode. Also known as ‘slave mode’. In this mode the payment app running on Albert is fully controlled by a specific third party app. The controlling app is typically an ECR or POS system. This mode allows the app to take full control over the payment app, specifying precise behaviour and influencing the transaction flow.

Notes:

  1. The payment APIs only work in standalone mode.
  2. The integrated mode is still in development, and is not currently available.
