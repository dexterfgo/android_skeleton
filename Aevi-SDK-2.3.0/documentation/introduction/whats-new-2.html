What's new in SDK 2.x.x

SDK Release 2.0 is a significant release that supports major changes in Albert’s hardened Operating System platform and underlying security architecture. These changes have been introduced following extensive feedback from both our direct customers and the Developer Community.

If you are already familiar with the AEVI SDK for Albert, this section will provide you with an overview of precisely what has changed. If you are new to the AEVI SDK, then you can skip the rest of this section.

Apps compiled using versions 1.x.x of the SDK are incompatible with the Albert running the new platform and all Albert’s will be migrated to this new platform. As a result, migration to SDK 2.x.x is mandatory for all Apps. The changes to the APIs are only minor and refactoring Apps should be reasonably quick and straightforward.

Type Enforcement

The original Applications Types that were provided in the 1.x.x releases of the AEVI-SDK (payment, vaan and vaas) have been retired. The new Application types are:

  • vaa_merchant
  • vaa_network

For further information on the features and capabilities of these new Application Types, please refer to Type Enforcement in the Platform Hardening, Customisations and Restrictions Section and Type Enforcement in the Permissions and Manifest Section.

Payment Application Sandbox Enhancements

The Sandbox in which the Payment Application runs has been further enhanced and hardened. As a result, some of the APIs have had to change. Further information is provided in the API changes subsection below.

Removal of the Trusted Communications Manager

The Trusted Communications Manager restrictions for Apps that require access to the Payment APIs have been removed. App developers are no longer required to use the transparent proxy provided by the TCM. Apps can use both secure and open protocols to access data and are free to use services that provide SHA-1 signed certificates. However, declaration of endpoints is still required in the AndroidManifest.xml.

Further information is available in the Permissions and Manifest Section

API Changes

A number of API changes have been implemented:

  • The PaymentConfiguration API is no longer supported. Users are required to use the PaymentAppConfiguration API instead. This exposes a similar set of methods but must be accessed via an Intent which replaces the original ContentProvider interface. Please review the Configuration & Status section for more details.
  • The PlatformCompatibility API has been removed. The methods have incorporated into the TerminalConfiguration and PaymentAppConfiguration APIs
  • The UriBuilder utility class has been removed as it is no longer required

Creation of Separate Simulators for the Device and Payment App

The original Simulator app has been split into two separate apps: AEVI Device Simulator simulates the device and ecosystem related APIs; AEVI Payment Simulator emulates the payment related APIs.

This change has been necessary to support the new platform. However, it provides additional benefits that will become apparent when we expand our products and customer base.

Migration Checklist

If you have an existing App that was compiled against the 1.x.x APIs, the following checklist should assist in migration to SDK 2.x.x

  • Change the App Type for your application in the App Manifest - If you were using vaas, please now use vaa_network. Please also note that vaa_network has no access to any of the AEVI SDK APIs. Therefore if your App needs access to these, please use the vaa_merchant App type
  • Migrate any code using the PaymentConfiguration API to the PaymentAppConfiguration API. Note that this API is based on Intents and therefore you will need to refactor this.
  • If you are using the PlatformCompatibility API, please refactor your code to use the methods in the TerminalConfiguration and PaymentAppConfiguration APIs
  • If your app is a Payment App, you will need to refactor your network access code to use standard Java network access APIs
  • Please test your App with the updated Emulator

The Development Lifecycle rules remain largely unchanged and we urge you to follow the guidelines contained in that section.
