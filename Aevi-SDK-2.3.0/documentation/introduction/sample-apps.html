Sample Applications

The AEVI SDK includes sample code for the Payment APIs. This illustrates how to work with the APIs and provides an indication of how the payment integration functions.

Please note that in order to run the examples, the AEVI simulators must be installed. Instructions to install are provided in the Installation Guide

Device and Payment Simulators

The Device and Payment Simulators are primarily background applications providing services to support app development for the AEVI platform. They implement all the APIs published by the SDK. For example, when an app uses the Payment API to process a payment, the Payment Simulator is called to complete the process.

The simulators also include some extra screens that allow both configuration and analysis of the behavior of an app. This is made available via the AEVI Device Simulator and AEVI Payment Simulator icons on the Launcher. When instantiated, they each present a menu with configuration options that are self-explanatory.

Device Simulator Main Menu

Configuration

The following configuration options are available on the Payment Simulator:

  1. Currencies
  2. Merchants
  3. Transaction types
  4. Integrated Mode

The following configuration options are available on the Device Simulator:

  1. View Shared Preferences
  2. View Crash Logs
  3. View Transaction Logs

Pressing any of these options will initiate a separate screen to set the configuration. The configuration is stored using the standard Android Shared Preferences functionality. The data can be cleared from the Settings > Apps screen, by navigating to the AEVI Payment Simulator info screen and pressing the Clear data button.

Differences between the Simulators and the real Device

The Device and Payment Simulators have been written primarily to enable a developer to test their App. They explicitly provide the ability to simulate failure conditions enabling developers to cater for all outcomes. Furthermore, they are completely standalone and therefore independent of the host systems typically required for processing payments.

The simulators’ screens mimic the functionality of real Albert screens whenever possible. However, the real device includes some UI customisations that are not available on generic Android devices. The one key exception is the option to cancel an activity: This can be achieved by pressing the back button on your android device. This will result in an “Are you sure?” message. Note that the Payment app on the real Albert device implements its own version of the Back Button. Furthermore, the Home and Menu buttons are also disabled.

Simulation of Integrated Mode

The Payment Simulator provides basic support for testing your App when the Device is in Integrated Mode:

  • A Switch is provided in the Payment Simulator to set Standalone or Integrated Modes
  • When in Integrated Mode, access to Financial Transactions is disabled
  • Broadcast Events are sent when the Simulator is switched between Standalone and Integrated Modes

The purpose of this example is to demonstrate how the payment app handles some different payment and refund transactions.

The start-up screen (see Figure 3 below) shows different sample activities, each demonstrating a part of the AEVI SDK.

Sample app start screen

The compiled version of this sample app can be installed using Android Debug Bridge (ADB):

ADB install <sdk>/samples/AeviSample-2.3.0.apk

The Smart Card and Mag Stripe examples require the use of an approved external USB card reader, plugged into the development tablet. See Card Reader API for more details. Running these samples for the first time, a conformation pop up will appear asking the user to allow the Device simulator to access the USB device. This pop-up confirmation dialog will not appear on the production Albert device.

USB confirmation pop-up

The Smart Card Sample sends APDU commands to read data from a non-payment JavaCard. It is recommended that the smartcard listed in the Smart Card API section is used. Although the Mag Stripe sample can read data from a payment card when using the simulator, this is disabled on the production hardware.

The “Printing”, “Smart Card”, and “Mag Stripe” samples all require additional hardware and so will not work on the Android emulator.

The “Secure Communications sample” requires a URL change (see Secure Communications) and so will only run without error on the Albert device or on the emulated Albert device.

To The Movies Network sample

To The Movies! Apps have been written to demonstrate the concept of an App that can both take a payment and communicate with an external service (over a secure channel). There are two versions of the app included in the SDK: A classic Java App; A web based App that uses typical web technologies (HTML5/JavaScript) running in a lightweight Java based container.

The classic version of this sample app can be installed using ADB:

adb install <sdk>/samples/ToTheMoviesClassic-2.3.0.apk

The web version of this sample app can be installed using ADB:

adb install <sdk>/samples/ToTheMovies-2.3.0.apk

The two screenshots below are from the classic version and show the lists of movies that are currently playing and the ‘buy ticket’ screen where a customer can read additional information and buy an entrance ticket. The movie DB API

Javascript bridge

In order to start an actual payment, the Javascript application needs to make a call to the Java part of the application. This is realized by exposing a JavaScript bridge that can be called from the code running in the browser.

This bridge exposes all the function available to the JavaScript part and looks as follows;

private class JavascriptBridge {

    @JavascriptInterface    
    public void buyTicket(String movieId, String amount) {
      PaymentRequest paymentRequest = new PaymentRequest(new BigDecimal(amount)));
      startActivityForResult(paymentRequest.createIntent(), Integer.parseInt(movieId));
    }

    @JavascriptInterface    
    public void printTicket(final String title) {
        ...
    }
}

The bridge is registered on start of the activity using the addJavascriptInterface method on a WebView instance.

webView.addJavascriptInterface(javascriptBridge, "Bridge");

Once this bridge has been registered the following JavaScript code can be executed;

Bridge.buyTicket(1234,"14.00");

This will call the Java code inside the buyTicket function which in turn will create a PaymentRequest that is send to the Payment Application using an Android Intent.

Caution! Due to a vulnerability identified in Android 4.0, the WebView component has been patched and all functions exposed to JavaScript must include the @JavascriptInterface annotation. Note that this annotation is only supported from Android API Level 17. Therefore, you are required to ensure that your compilation settings are set accordingly and that you have the corresponding build tools installed within your environment.

For sample configuration settings for Eclipse or Gradle, please refer to the ToTheMovies Sample source code.

Payment results

When the Payment app returns control to our app, onActivityResult is called with an Intent that contains information about the status of our transaction. A TransactionResult object can be extracted from this intent using the TransactionResult.fromIntent method.

We can use this information to navigate to success or failure page within the HTML part of the application.

// handle the payment app response
@Override
protected void onActivityResult(int requestCode, int resultCode, Intent data) {
super.onActivityResult(requestCode, resultCode, data);

TransactionResult transactionResult = TransactionResult.fromIntent(data);

switch (transactionResult.getTransactionStatus()) {
  case APPROVED:
    javascriptBridge.navigate("/success/" + requestCode);
    break;
  default:
    javascriptBridge.navigate("/failure");
    break;
}

The navigation is done using ‘client-side’ routing (aka hash-routing). This allows us to specify the state of the application using the hash part of the URL.

Below are two examples of different application states based on the hash part of the URL;

http://example.com/index.html#success
http://example.com/index.html#failure

This information can be captured in the browser and a success or failure page can be displayed accordingly.

Type Enforcement and Firewall Rules

Before the app can communicate with external servers hosting the REST API and images, the hostnames of these servers must be declared in the Android Manifest file. Along with this hostname declaration, the app must be declared as a vaa_merchant type and request the correct permission.

These configuration entries can be made by altering the manifest tag as follows;

<manifest xmlns:android="http://schemas.android.com/apk/res/android"
          package="com.aevi.tothemovies">

  ...

  <uses-permission android:name="android.permission.INTERNET" /> 

  <meta-data android:name="application-type" android:value="vaa_merchant"/>
  <meta-data android:name="ip-addresses" android:value="443|api.themoviedb.org|10000"/>
  <meta-data android:name="ip-addresses" android:value="443|d3gtl9l2a4fn1j.cloudfront.net|10000"/>

</manifest>

Please see the Network Access section for more details on the notation.
